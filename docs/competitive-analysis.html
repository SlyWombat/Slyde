Where Slyde sits in the open-source photo-frame landscape

A fact-checked map of free / self-hostable digital photo-frame software — and the one niche almost nobody else occupies: reviving discontinued commercial frame hardware over a reverse-engineered LAN protocol, fed read-only from your own Immich library.
🔎 5 search angles 📄 19 sources fetched 🧪 25 claims · 3-vote adversarial verify ✅ 25/25 confirmed · 0 killed 🏷️ 13 projects compared

The one-paragraph verdict

As of 2026 the open-source frame space is crowded on “render a slideshow on a Pi, browser, or general-purpose display,” and Immich-sourcing is now table stakes (ImmichFrame, immich-kiosk, espframe, syno-photo-frame, Magic Frame all do it). It is nearly empty on Slyde’s actual turf:

  • Reviving closed, non-Android frame hardware via a reverse-engineered LAN protocol. The only hardware-control peers target Frameo frames — which run Android and expose official ADB, so they aren’t reverse-engineering anything, and have near-zero adoption.
  • An emulator and a real OTA firmware/app pipeline — no surveyed competitor ships either.
  • One-way, read-only Immich sourcing as an explicit safety guarantee.

Competitors are genuinely stronger on slideshow configurability, breadth of deployment targets, and community momentum. Memento’s least-differentiated feature is the Pi “soft-frame” mode — that quadrant is busy.

3 The field, by cluster

🟦 Immich renderers — the crowded core

ImmichFrame (GPL-3.0) is the gravity well: 2.1k★, 43 releases, 1,182 commits, latest v1.0.33.0 (Apr 10 2026), Immich-exclusive, runs in a browser / on devices. immich-kiosk (AGPL-3.0) is the “highly configurable” browser-kiosk alternative. syno-photo-frame (GPL-3.0, Rust) is the lightweight native-on-Pi choice (Immich + Synology via shared-album links), deliberately avoiding a browser to run on a Pi Zero 2. Magic Frame adds Immich + WebDAV in a Docker dashboard.

Takeaway: Immich-sourcing is no longer a feature you can win on — it’s the entry ticket. Memento already has it; the edge has to come from elsewhere.

🟧 Hardware-revival peers — thin, and not what they look like

The only projects that control a commercial frame target Frameo: phito/frameo-api (Go; screen on/off, brightness, next/prev swipe — no photo sourcing at all) and maju6406/frameo-control-api (GPL-3.0; ADB over USB/WiFi, direct POST /upload, 2 stars, no OTA, no library integration).

The crucial distinction: Frameo frames run Android and Frameo officially documents ADB. These tools ride a vendor-supported debugging path — they explicitly “leverage standard Android debugging rather than reverse-engineering proprietary protocols.” Memento’s frame is a closed, non-Android device whose LAN protocol had to be reverse-engineered from the dead official app. That is a categorically harder and rarer capability — and as far as the survey found, no open-source project revives non-Android commercial frames (Pix-Star, Nixplay, Aura, Skylight, Electric Objects/Memento) this way.

🟥 Legacy, art, and WebDAV — adjacent but not substitutes

mrworf/photoframe (GPL-3.0) was the canonical Pi frame, but its primary source — Google Photos — is now broken by API changes; only Pexels + USB remain, no Immich. OpenFrame (GPL-3.0, Node.js) is an elegant idea — push digital art (images/video/web/GLSL shaders) to Pi frames from a web app — but it’s art-focused, cloud-coordinated, and effectively abandoned: the controller’s last commit was Jan 2021 and its self-host API server hasn’t been touched since 2017. OpenPhotoFrame and photOS serve the Nextcloud/WebDAV crowd; neither speaks Immich.

OpenFrame is also the cautionary tale: a cloud-coordinated frame platform whose own coordination server is dead is one DNS lapse from the exact fate that bricked Memento frames — which is precisely the failure mode Slyde is built to cure (local-only, no cloud dependency).

4 Where Memento wins · where rivals win

✅ Slyde is differentiated on

  • Reviving closed/non-Android frame HW via a reverse-engineered LAN protocol — no verified open-source equivalent.
  • OTA firmware/app pipeline (md5-verified bundles, GitHub-release driven) — no surveyed competitor ships OTA.
  • A faithful software emulator — unique in the field; the whole test suite runs against it.
  • One-way, read-only Immich sourcing as an explicit, designed-in safety contract.
  • Manager↔device split with per-frame canvas/orientation, smart fit, background sync jobs — a real management plane, not just a renderer.

⚠️ Competitors are stronger on

  • Slideshow configurability & polish — immich-kiosk/ImmichFrame have years of transitions, layouts, weather/clock overlays.
  • Breadth of deployment targets — browser, any TV, NAS, Pi Zero, ESP32; Memento is narrower.
  • Community momentum — ImmichFrame’s 2.1k★ and release cadence dwarf this niche.
  • The Pi “soft-frame” mode itself — Memento’s least-differentiated feature; syno-photo-frame, immich-kiosk and Magic Frame already own “render on your own screen.”
  • Source breadth — several support Synology/WebDAV/art/cloud in addition to Immich.

5 Recommendations & roadmap

Lean into the empty quadrant; don’t try to out-slideshow ImmichFrame.
Do now

Own the “frame-revival” story explicitly

Lead the README/positioning with “revive a dead commercial frame, no cloud, no e-waste.” That’s the unique, defensible claim. Publish the reverse-engineered docs/protocol.md prominently — it’s a moat and a community magnet that no Immich-renderer can copy.

Do now

Make “read-only Immich” a headline guarantee

Open question from the research: do ImmichFrame/immich-kiosk write back to Immich? Position Memento’s one-way contract as a trust feature for people nervous about a frame touching their library.

Do next

Generalize the protocol layer into a “frame-revival toolkit”

The architecture (core protocol + manager + emulator) is the real asset. Document how to add a second dead-frame protocol. Becoming “the OpenWrt of abandoned smart frames” is a category nobody owns.

Do next

Turn OTA + emulator into visible selling points

Both are unique. A short demo (emulator → “update available” → frame self-applies) shows capability competitors can’t match. Cut a softframe-v0.1.1 to make the update path tangible.

Later

Borrow the table-stakes polish selectively

Add the slideshow niceties users expect (transition styles, Ken-Burns, clock/weather overlay) so the soft-frame mode isn’t strictly behind syno-photo-frame/immich-kiosk — but treat it as parity work, not the differentiator.

Later

Court the Frameo/Pix-Star/Nixplay refugees

Each discontinued or cloud-locked frame is a potential Memento target. A protocol-contribution guide + a “which frames are supported” page could pull in the jailbreak communities (e.g. nixplaysucks) already trying to do this by hand.

7 Caveats & open questions

Caveats

  • The emulator/OTA differentiation rests on absence of evidence across competitor docs, not explicit denials — strong, not absolute.
  • Some named projects (Immich’s own native slideshow/cast, MagicMirror² photo modules, DAKboard, PiFrame, and non-Frameo revival tools) weren’t covered by surviving claims and are absent from the matrix.
  • “Reverse-engineered protocol” differentiation is proven specifically vs Frameo (Android/ADB); not tested against hypothetical non-Android revival projects (which may not exist in OSS).
  • ImmichFrame’s “Apr 10” release date required cross-checking GitHub’s year-omission convention to confirm 2026 — high confidence, time-sensitive.

Open questions worth chasing

  • Does Immich’s own native slideshow/cast (or a separate immich-frame) overlap with Memento, and how configurable is it?
  • Any OSS reviving non-Android commercial frames (Pix-Star, Nixplay, Aura, Skylight, Electric Objects) — Memento’s truest competitors?
  • Do ImmichFrame / espframe ship real OTA beyond container/release updates?
  • Do ImmichFrame/immich-kiosk write back to Immich, or are they also effectively read-only? (Sharpens Memento’s read-only claim.)
